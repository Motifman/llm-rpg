# 実験計画の書き方

この文書は、LLM 実験を回す前に「何を見たい run なのか」を残すための運用メモです。設定入力経路の仕様は [docs/experiment_config_management.md](./experiment_config_management.md) を参照してください。

## 目的

実験の目的は、毎回きれいな比較で差分を証明することだけではありません。実 LLM run は費用がかかり、挙動も確率的に揺れるため、明確な基準 run を置けない探索や観察もあります。

それでも、run 前に目的と観測したいことを書いておくと、あとから「なぜこの条件で回したのか」「結果から何を読めるのか」「次に何を変えるべきか」を判断しやすくなります。

## run の型

### 探索 run

まだ比較対象を決める前に、挙動の輪郭を見る run です。

- 目的: 新しい機能や profile がどんな挙動を生むかを見る
- 基準 run: なくてもよい
- 変える値: 複数あってもよいが、あとから読めるように理由を書く
- 成功条件: 例外なく完走、trace が出る、見るべき場面が起きる、など

### 観察 run

普段使いの標準設定で、質感や失敗の出方を見る run です。

- 目的: エージェントの質感、停滞、協調、記憶の働き方を見る
- 基準 run: 必須ではない
- 変える値: 原則 profile に固定し、場面や seed だけを変える
- 成功条件: 分析対象になる十分な trace が残ること

### 比較 run

ある機能や設定の差分を見る run です。

- 目的: 変えた設定が観測にどう影響するかを見る
- 基準 run: ある方がよい
- 変える値: 可能なら 1 つに絞る
- 成功条件: 比較したい trace / 指標が両方の run に存在すること

比較 run でも、実 LLM 費用の都合で厳密な反復や完全な交絡除去ができない場合があります。その場合は「比較できること」と「比較できないこと」を計画に明記します。

## run 前に書くこと

最低限、次の項目を残します。

- run id: `var/runs/...` に使う名前
- run の型: 探索 / 観察 / 比較
- profile: 使う `data/experiment_profiles/*.json`
- scenario: 使うシナリオ
- 目的: 何を見たいのか
- 注目する場面: どの出来事や tick 付近を見るか
- 必須 trace / 指標: 結果判断に必要な観測点
- コスト上の制約: tick 数、並列度、反復数を絞る理由
- 比較対象: ある場合だけ書く。ない場合は「なし」と明記する
- この run だけで言えること / 言えないこと

## 計画例

```markdown
# v3coop_stagnation_probe_001

- run の型: 観察
- profile: belief_goal_full
- scenario: data/scenarios/survival_island_v3_coop.json
- 目的: 停滞感が自己観測と次行動にどう出るかを見る
- 注目する場面: 同じ行動を繰り返した後の reflect / reasoning / 行動選択
- 必須 trace / 指標:
  - AGENT_REASONING_ENGAGED
  - BELIEF_EVIDENCE
  - EPISODIC_RECALL
  - LLM call metrics
- コスト上の制約: まず 1 run だけ。反復は異常や有望な挙動が見えた後に判断する
- 比較対象: なし
- この run だけで言えること:
  - 配線が生きているか
  - 観察した場面で停滞感が表出したか
- この run だけで言えないこと:
  - 停滞 reasoning の平均的な効果
  - 他 profile との差分の有意性
```

## 読み方の注意

- 基準 run がない run を失敗扱いしない。探索 run と観察 run は、次に何を見るかを決めるための材料でもある。
- 比較 run でない結果から、機能の効果を強く断定しない。
- コストを理由に反復しない判断は正当。ただし、その run だけで言えないことを明記する。
- run 後の分析では、計画に書いた必須 trace / 指標が実際に出ているかを最初に確認する。
