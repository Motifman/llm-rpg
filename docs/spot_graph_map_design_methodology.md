# spot グラフのマップ設計方法論

## 目的

spot グラフのシナリオを、手書きでも将来の LLM 生成でも破綻しにくく作るための方法論を定める。

現状の `survival_island_v3_coop` は拠点を中心に放射状へ広がるほぼ木構造で、要所への経路が単一になりやすい。この形だと「どの道を選ぶか」「危険を避けて迂回するか」「過去に通った場所をどう覚えるか」が生まれにくい。将来、being に方向・距離を伴う空間記憶を持たせるためにも、地理を先に定め、その地理から接続を導く作り方へ寄せる。

この文書は方法論とバリデータ設計だけを扱う。`position` スキーマ、バリデータ、viewer の座標駆動、既存シナリオへの適用は、承認後に別 PR で実装する。

## 現状の前提

- spot には地理座標が無い。
- `connections` は明示的な辺リストで、`from` / `to` / `travel_ticks` / `is_bidirectional` / `passage` を持つ。
- 条件付き通行は `reactive_bindings.passages` で `passage` 状態を切り替える。
- trace viewer は `connections` だけを読み、Cytoscape の `cose` 自動配置で描画している。表示上の配置は物理地理ではない。
- 既存シナリオの辺には、移動コストや条件付き通行の意図が既に埋まっている。改修時に既存辺を消すと、過去のバランスやテストの意味が崩れる。

## 基本原則

### 1. 空間の正本は spot の `position`

最初に島・建物・洞窟などの地理を決め、各 spot に `position` を置く。接続はその座標と地形から導く。

`connections` は正本ではなく、「この地理で実際に通れる道を採択した結果」とみなす。

### 2. 既存マップ改修は「背骨維持 + 追加」

既存シナリオを改修するときは、現行の辺を原則として消さない。

手順:

1. 現行の辺でつながっている spot 同士が近くなるように座標を置く。
2. 既存の `travel_ticks` と地理距離が大きく矛盾しないよう調整する。
3. 近接しているが未接続の spot ペアを候補に出す。
4. 地形・崖・水域・山稜・物語上の制約で候補を落とす。
5. 残った候補を人間が採択し、閉路と迂回路を足す。

これにより、過去の意図を壊さず、木構造からメッシュ構造へ寄せる。

### 3. 接続は「近接候補 + 地形ガード + 採択」

座標が近いだけでは接続しない。水域、崖、密林、崩落、急斜面などの通行障害を見て、通れる道だけを採択する。

近いが通れない例:

- 入り江を挟んで対岸にある。
- 崖の上下にあり、道や梯子が無い。
- 山頂と麓が直線距離では近いが、急斜面で直登できない。

遠いが接続してよい例:

- 海岸沿いの長い道。
- 尾根道。
- 川沿いの緩い登り。

### 4. 木ではなく閉路を持つ

主要な移動判断が生まれるには、少なくとも一部に閉路が必要。

必須方針:

- グラフ全体が木だけで終わらない。
- 重要地点には原則として 2 経路以上で到達できる。
- ただし物語上の関門は維持する。目標を無効化する近道は張らない。
- 行き止まりは意図がある場合だけ許す。例: 貴重資源、危険、伏線、隠し場所。
- `travel_ticks` は座標距離と地形負荷を反映する。
- 既定は双方向。片方向は崖を下る、崩落する、扉が閉まるなど明確な理由がある場合だけ使う。
- 条件付き通行は辺の有無を消すのではなく `passage` で表現する。

### 5. viewer は座標があれば座標駆動

`position` があるシナリオでは、viewer は `position` を使って初期配置する。`position` が無いシナリオでは従来の `cose` 自動配置へ戻す。

これにより、分析時の地図が物語上の地理と一致する。自動配置は見た目の都合で辺を配置するだけなので、空間認知の検証には使わない。

## `position` スキーマ案

後方互換のため、`position` は任意フィールドにする。

```json
{
  "id": "beach_camp",
  "name": "浜辺の拠点",
  "category": "FIELD",
  "position": {
    "x": 0,
    "y": 0
  }
}
```

座標の意味:

- `x`: 東西方向。正を東とする。
- `y`: 南北方向。正を北とする。
- 単位はシナリオ内の相対距離。メートル等の物理単位に固定しない。
- 将来 `z` が必要になったら追加する。初期案では `x` / `y` の 2 次元に絞る。

任意の top-level メタ情報:

```json
{
  "map_geometry": {
    "position_schema_version": 1,
    "distance_unit_label": "relative",
    "recommended_connection_distance": 3.0,
    "max_direct_connection_distance": 5.0,
    "key_spots": ["beach_camp", "signal_peak"],
    "barriers": [
      {
        "id": "west_cliff",
        "kind": "cliff",
        "polyline": [
          {"x": -2, "y": 1},
          {"x": -2, "y": 5}
        ],
        "allows_crossing_tags": ["trail", "rope"]
      }
    ]
  }
}
```

`barriers` は初期実装では任意。最初は「座標と辺の距離整合」「到達性」「閉路」「重要地点の冗長経路」を先に検査し、バリア交差検査は第 2 段階でもよい。

## マップ作成手順

### 手書き作者向け

1. 舞台の地理を文章で決める。
   - 例: 南に浜、中央に森、北東に山、北西に崖、東に川。
2. 主要地点を列挙する。
   - 拠点、資源、危険地帯、目標地点、関門、避難場所。
3. 各 spot に `position` を置く。
4. 近接している spot ペアを候補辺として出す。
5. 地形上通れない候補を落とす。
6. 既存の背骨になる道を残す。
7. 重要地点に 2 経路以上あるか確認する。
8. 近道が勝利条件や関門を壊していないか確認する。
9. `travel_ticks` を距離と地形負荷で調整する。
10. バリデータを通し、警告を見て座標か接続を修正する。

### LLM 生成向け

LLM にシナリオを生成させる場合は、JSON を一気に書かせない。次の順に段階出力させる。

1. 地理説明を作る。
2. spot 一覧と `position` を作る。
3. 近接候補辺を列挙する。
4. 候補ごとに「採択 / 却下」と理由を書く。
5. 採択辺から `connections` を作る。
6. 重要地点の 2 経路性、閉路、到達不能、チョークポイントを自己検査する。
7. バリデータを通す。
8. バリデータ警告を反映して再生成する。

生成指示テンプレ案:

```text
まず地理を決めてください。次に spot ごとの position{x,y} を決めてください。
connections は position から導出し、近接していても水域・崖・山稜などで通れない辺は採択しないでください。
グラフは木にせず、主要目標 spot には 2 経路以上で到達できるようにしてください。
ただし勝利条件を無効化する近道は作らないでください。
最後に、到達不能 spot、閉路の有無、重要地点への冗長経路、意図しない行き止まり、travel_ticks と距離の不整合を自己検査してください。
```

## バリデータ設計

### 入力

- scenario JSON
- 任意の検査設定
  - `key_spots`
  - `allowed_dead_ends`
  - `max_direct_connection_distance`
  - `distance_to_tick_ratio`
  - `strict`

`key_spots` は `map_geometry.key_spots` から読む。指定が無ければ警告だけにする。

### 出力

機械可読な結果を返す。

```json
{
  "ok": false,
  "errors": [
    {
      "code": "KEY_SPOT_SINGLE_ROUTE",
      "severity": "error",
      "message": "signal_peak は beach_camp から 1 経路しかありません",
      "spots": ["beach_camp", "signal_peak"]
    }
  ],
  "warnings": [
    {
      "code": "LONG_EDGE_DISTANCE",
      "severity": "warning",
      "message": "beach_camp -> signal_peak は座標距離に対して travel_ticks が短すぎます",
      "connection": "beach_camp->signal_peak"
    }
  ],
  "metrics": {
    "spot_count": 8,
    "undirected_edge_count": 10,
    "cycle_rank": 3,
    "dead_end_spots": ["west_cave"],
    "articulation_spots": ["forest_crossroads"]
  }
}
```

### 検査項目

| 検査 | 既定 | 実験 strict | 内容 |
|---|---:|---:|---|
| JSON 構文 | error | error | scenario が読めない場合は即失敗 |
| `position` 型 | error | error | `x` / `y` が数値でない spot を弾く |
| ID 参照整合 | error | error | connection / passage / key_spots が存在しない spot を参照していないか |
| 到達不能 spot | warning | error | 無向グラフとして到達不能 spot が無いか |
| 木構造 | warning | error | `cycle_rank == 0` なら閉路無し |
| 重要地点の 2 経路性 | warning | error | 拠点から key_spots へ辺または spot を 1 つ失っても到達可能か |
| チョークポイント | warning | warning | articulation spot を列挙する。意図的か確認する |
| 行き止まり | warning | warning | degree 1 の spot を列挙し、許可リスト外なら警告 |
| 長すぎる直接辺 | warning | warning | 座標距離が閾値を超える connection |
| 近接未接続候補 | info | warning | 近いのに接続されていない spot ペアを候補として出す |
| `travel_ticks` 距離整合 | warning | warning | 距離に対して移動 tick が極端に短い / 長い |
| バリア交差 | warning | error | `barriers` がある場合、許可タグなしに横切る辺を検出 |
| 目標無効化の近道 | 手動 | 手動 | win condition の意味を壊す辺は機械だけでは判断しない |

### 2 経路性の判定

初期実装では、拠点から各 `key_spots` へ次を調べる。

- 各 connection を 1 本ずつ除去しても到達できるか。
- 各中間 spot を 1 つずつ除去しても到達できるか。

これで「橋が 1 本落ちると山頂へ行けない」「森の分岐点が唯一の関門になっている」などを検出できる。

条件付き通行の扱いは 2 段階に分ける。

1. 初期実装: `connections` の構造だけを見る。
2. 後続: `passage` の既定状態と代表的な scenario state を与え、状態別に到達性を検査する。

### 警告と失敗の使い分け

- ローカル設計時: warning 中心。作者が意図を見ながら直す。
- 継続的検査: 既存シナリオは warning、明示的に `strict` を付けたシナリオだけ error。
- 実験モード: `strict` を有効にし、到達不能・閉路無し・重要地点単一路は fail-fast。

この段階分けにより、既存シナリオを一度に壊さず、新しい実験シナリオだけ品質を強く要求できる。

## viewer 連携

viewer は次の順で配置方式を選ぶ。

1. 全 spot に `position` がある: 座標駆動で配置する。
2. 一部だけ `position` がある: 警告を出し、従来の自動配置へ戻す。
3. `position` が無い: 従来の自動配置。

座標駆動時は `x` / `y` の範囲を画面サイズへ正規化し、`travel_ticks` や `passage.kind` を線の太さ・破線・色へ反映できるようにする。

## toy 例

地理:

- 南に浜辺の拠点。
- 中央に森の分岐。
- 東に小川。
- 北に山道。
- 北東に狼煙台。
- 西に洞窟。

座標:

```json
{
  "spots": [
    {"id": "camp", "name": "浜辺の拠点", "position": {"x": 0, "y": 0}},
    {"id": "forest", "name": "森の分岐", "position": {"x": 0, "y": 2}},
    {"id": "stream", "name": "小川", "position": {"x": 2, "y": 2}},
    {"id": "ridge", "name": "尾根道", "position": {"x": 1, "y": 4}},
    {"id": "signal", "name": "狼煙台", "position": {"x": 2, "y": 5}},
    {"id": "cave", "name": "西の洞窟", "position": {"x": -2, "y": 2}}
  ]
}
```

接続図:

```text
        狼煙台
        /   \
     尾根道  小川
       |    /
 西の洞窟 - 森の分岐
          |
       浜辺の拠点
```

この例では、狼煙台へ「浜辺の拠点 → 森の分岐 → 尾根道 → 狼煙台」と「浜辺の拠点 → 森の分岐 → 小川 → 狼煙台」の 2 経路がある。西の洞窟は行き止まりに見えるが、資源や危険を置く意図があれば許可できる。

## `survival_island_v3_coop` への適用方針

実座標はこの文書の承認後、人間が地理解釈を決めて割り当てる。

適用時の方針:

- 現行の辺は削除しない。
- 浜辺・拠点・森・小川・山道・狼煙台の大まかな位置関係を先に決める。
- 山頂 / 狼煙台は一本道にしない。ただし救助条件を無効化する直通近道は作らない。
- 資源 spot は複数経路から寄れるようにし、採取・空腹・危険回避の判断を作る。
- 危険地帯や高コスト地形は `travel_ticks` と `passage` で表現する。

## 空間認知への接続

`position` が入ると、being は将来次のような経験を持てる。

- 「拠点の北に森がある」
- 「狼煙台へは尾根道経由と小川経由がある」
- 「夜に山道を通るより、小川沿いに迂回した方が安全だった」
- 「前回、洞窟は行き止まりだったが食料があった」

これはハードコードされた経路知識ではなく、観測と移動経験から育つ地図記憶の土台になる。座標は viewer の見た目だけでなく、being の空間学習を検証可能にするための正本として扱う。

## 実装分割案

1. `position` スキーマの読み込みと保存
   - scenario loader
   - DTO / SQLite codec / snapshot への影響確認
   - 後方互換: 未指定なら従来通り
2. バリデータ
   - 到達不能
   - 閉路
   - 重要地点 2 経路性
   - degree / チョークポイント
   - 距離と `travel_ticks`
3. viewer 座標駆動
   - `position` が全 spot にあれば座標配置
   - 無ければ従来 fallback
4. `survival_island_v3_coop` への適用
   - 人間レビューで地理解釈を確定
   - 座標追加
   - 横断辺追加
   - バリデータを strict で通す
5. LLM 生成支援
   - 生成指示テンプレ
   - バリデータ結果を再生成入力へ戻す流れ

## 未決事項

- `position` を `spots[].position` に置くか、top-level `map_geometry.positions` に分離するか。
  - 初期案は `spots[].position`。spot の地理属性として読む方が自然で、viewer も扱いやすい。
- `barriers` を最初から入れるか。
  - 初期案は任意。座標と接続品質の検査を先に入れ、バリア交差は後続でもよい。
- 2 経路性を「辺を 1 本失っても到達」までにするか、「spot を 1 つ失っても到達」まで要求するか。
  - 初期案は両方を計算し、error にする対象は `key_spots` ごとに設定可能にする。
- `travel_ticks` の推奨式。
  - 初期案は距離から推奨範囲を出して warning。地形負荷や危険度は作者判断を残す。
