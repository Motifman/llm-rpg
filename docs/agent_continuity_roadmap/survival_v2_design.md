# 漂流島 v2 設計: 4人ペルソナ × 動的連合 × 14日

> 親ドキュメント: `docs/agent_continuity_roadmap/README.md`
>
> このドキュメントは v2 シナリオの **仕掛けと制約** を定義する。
> ハードコードされた「Day N で X が起きる」シナリオ進行は**意図的に書かない**。
> 何が起きるかは、置かれた仕掛けと制約からエージェントの判断で創発する。

---

## 0. 大原則 (これを破ったらこのドキュメントの目的が崩壊する)

### エージェントの行動と選択をスクリプトに書かない

字面で「Day N に何かが起きる」を全部禁止する話ではない。
**世界の出来事**はスクリプト可、**エージェントの選択**は不可、という分け方。

| OK (世界の側で起きる出来事 / 制約) | NG (エージェントの行動と選択の決め打ち) |
|---|---|
| Day 3 に嵐が吹く | "Day 5 にリオが隠し食いをする" |
| 食料は採取から N tick で腐る | "3対1 の対立構図が生まれる" |
| Hunger は tick 毎に増える | "Day 8 にエイダが薬を諦める" |
| 救助船は tick 70 / 140 で通過する | "Day 7 の救助はすれ違う" |
| カイの祖父の手記は洞窟の特定 spot にある | "カイが Day 6 に手記を発見する" |
| 夜間にモンスターが出現する | "ノアが Day 9 に指揮を奪われる" |

シナリオ JSON は **世界の状態空間 + 状態遷移ルール + 環境イベント + 終局条件**
までを書く。エージェントが何を選び、誰と組み、何を語るかは出力。
「行動はこうなるだろう」を **織り込んだメカニクス設計はしない**。

### 生存は全てに先立つ

各ペルソナの system prompt 冒頭に固定:

```
最優先事項: あなた自身が生き残ること。空腹・脱水・負傷・寒さは全て死に
つながる。全ての判断はこの最優先事項から導出される。
他の目標 (救助・秘密の達成・人間関係) は、生存できる前提でのみ意味を持つ。
```

これに加えて各ペルソナ固有の「第二の動機」(救助・遺品探し・帰宅・名誉) を
重ねる。第二の動機が生存と衝突したら、本来は生存を取る — でも LLM は本能を
持たないので、prompt と needs system の閾値メッセージで「死の予感」を再現する。

## 1. 4人ペルソナ

公開プロフィールと秘密の動機を分離する。秘密は本人の system prompt にのみ
書き、他者の prompt には書かない (初期状態の info_asymmetry)。

### エイダ (40代・医師)

- **公開**: 冷静沈着、合理判定が早い。キノコ・薬草を識別できる
- **第二の動機**: 過去のトラウマがある (誰を救うか迫られた経験)
- **秘密**: 救急用品の薬は1人分しか効かない。これを1人だけ知っている
- **話し方**: 落ち着いた敬体。「〜ですね」「〜と思います」

### ノア (20代・元自衛官)

- **公開**: 体力と判断力に自信、リーダー指向
- **第二の動機**: 全員を生還させて自分の指揮実績にしたい
- **秘密**: 実は除隊歴がある (理由は本人 prompt に記述)。リーダー失格を恐れる
- **話し方**: きびきびした常体。「やる」「行くぞ」

### リオ (30代・建築技師)

- **公開**: 寡黙で実直、手が早い。工作と釣りが得意
- **第二の動機**: 早く帰りたい (結婚式が待っている)
- **秘密**: 結婚前夜だった。焦りで他者を犠牲にする判断を取りうる
- **話し方**: 短い文。「悪い」「やっとく」

### カイ (18歳・留学生)

- **公開**: 陽気、動きが速い、観察力がある
- **第二の動機**: この島の地理に「不思議と詳しい」(他者には説明しない)
- **秘密**: 祖父がこの島で行方不明になっている。遺品 (古い羅針盤・手記) を
  探している。観測拠点跡 (廃屋) の存在を知っている
- **話し方**: 軽い口調。「マジで」「いいよー」

### ペルソナ間関係性 (初期状態)

全員、ツアー船で**初対面**。これにより:
- 初期信頼度は中立
- 関係性は 14 日のうちに**観測した行動だけ**から構築される (witness 必須)
- 後から発覚する秘密が信頼を上書きする

## 2. 世界の前提

### 漂着の経緯 (公開設定)

- 観光ツアー船 (日帰りクルーズ) が嵐で難破
- 4人は救命ボートで近くの無人島に流れ着いた
- 救命ボートは沖に流された (戻る手段はない)
- 14 日以内に救助される必要がある (期限の根拠は本文では言及しない)

### 観測拠点跡 (隠し設定)

- 30 年前、ある研究機関が島に観測拠点を設けていた
- カイの祖父はそこで働いていた最後の連絡を絶ったまま行方不明
- 観測拠点跡は廃屋として島に残っている
- カイだけがこの事実の一部を知っている (祖父の話を聞いたことがある)

これは **シナリオの仕掛けにのみ反映** され、明示的な「目標」にはしない。
カイがこの情報をどう扱うかは本人の判断 (= LLM の選択) に委ねる。

## 3. 世界ライフサイクル (6つの周期)

各周期は独立に進行し、相互作用する。

| 周期 | 進行 | 影響 |
|---|---|---|
| **昼夜** | 12 tick (昼8 / 夜4) | 夜は視界悪化、寒さで疲労増加、夜行性モンスター |
| **週** | 7日 = 84 tick | 累積疲労 (基盤側で別途調整) |
| **潮汐** | 30 tick (満/干) | 干潟・洞窟内部の通行可否 |
| **天候** | 確率遷移 (CLEAR ↔ RAIN ↔ STORM) | 嵐で岩礁・崖見張り台への passage 封鎖 |
| **救助船** | tick 60〜80 / 130〜140 で通過判定 (確率) | 狼煙が灯っていれば気づく |
| **腐敗** | 採取後 N tick で食料が腐る | 生魚 8, 焼魚 20, 果実 15, 貝 6, 真水 ∞ |

実装は別 Phase で順次足す (§7 参照)。

## 4. アイテム依存グラフ

### 設計方針

- **生存の根幹リソース** (水・食料) は再生するが、4 人分を維持するには採取を
  分担する必要がある
- **道具は希少だが致命的に単一ではない** (バックアップ手段は弱いがある)
- **フレーバーアイテム** (羅針盤の刻印、古い手記、写真) で物語の奥行きを足す
- **クラフトは浅く** — 焚き火点火・調理・狼煙の 3 つで充分。深い合成ツリーは
  足さない

### 現行 → v2 の変更案

#### 道具の冗長性

| アイテム | 現行 | v2 案 | 理由 |
|---|---|---|---|
| 骨ナイフ | 1 個 (船倉) | **2 個** (船倉 + 墜落機) | 蔓切り・キノコ切り分け・釣り針作成に必須。1 個だと隠匿されたら詰む |
| 釣竿 | 1 個 (洞窟) | **1 個 + 代替手段** | 洞窟は満潮で水没。代替で「即席の釣り糸」(蔓+骨ナイフ) を許可 |
| 火打石 | 2 個 (隠し入江 + 墜落機) | 現行維持 | 焚き火・狼煙の両方に必要、冗長性が要る |
| 救急用品 | 1 個 (墜落機) | 現行維持 | 1 個しかない事に意味がある (誰に使うかの倫理) |

#### 救急用品の効果 (新)

- `consume_effect`: HP +50, 出血状態解除 (status effect の付与/解除は要確認)
- 単発消費型
- 「自分に使うか、誰かに使うか」の判断が成立

#### オブジェクト接触ダメージ (既存メカニクスの活用)

新規スポット要素として:

- **岩礁海岸の崖**: `examine` 等で APPLY_DAMAGE (小: HP-5、滑落)
- **沼地のぬかるみ**: `cross` で APPLY_DAMAGE (小: HP-3、寒さ + 疲労)
- **廃屋の崩れた梁**: `enter` で APPLY_DAMAGE (中: HP-15、確率発生)
- **トゲのある実**: 採取で APPLY_DAMAGE (微: HP-1)

これらは「罠」ではなく**環境の自然な危険性**として配置する。

#### フレーバーアイテム (新規追加候補)

ゲーム上の効果ゼロ、物語のための小物:

| アイテム | 場所 | 物語的意味 |
|---|---|---|
| 古い羅針盤 (現行) | 洞窟内 | 文字盤に "SE" — 30 年前の観測拠点を示す? |
| 祖父の手記 (前半) | 廃屋の机 | 観測拠点の業務日誌の一部、署名は読めない |
| 祖父の手記 (後半) | 沼地奥の小屋 | 「家族に会いたい」で途絶える |
| 結婚式の招待状 | リオの初期所持 | 日付がツアーの翌週 |
| 軍の徽章 (除隊済) | ノアの初期所持 (隠し) | 認識票が削られている |
| 写真 (薬瓶ラベル) | エイダの初期所持 | 試験中であることを示す印 |
| 海図の切れ端 | 浜辺の流木に絡まる | 島の位置が記されていない |

これらは LLM が拾って読み、prompt に反映され、長期記憶テストに使われる。

### v2 アイテム依存グラフ (抜粋)

```
[流木] (4tick再生)        ──┐
[枯葉] (1個×2、初期固定)  ──┤
[火打石] (1個×2、初期固定) ┴→ build_fire → 焚き火(lit) ──┐
                                                          │
[骨ナイフ] (1個×2、船倉/墜落機)──┐                       │
[蔓] (12tick再生、要骨ナイフ) ────┘──→ [蔓ロープ]         │
[蔓ロープ] + [流木] ──→ build_shelter (避難所)             │
[蔓ロープ] + [骨ナイフ] ──→ make_makeshift_rod (即席釣竿)  │
                                                          │
[釣竿 or 即席釣竿] ──→ fishing ──→ [生魚]                 │
                                                          │
[生魚] + 焚き火(lit) ──→ cook_fish → [焼き魚] ←──────────┘
[生魚] (8tick後) → 腐敗 (食用不可 / 食べると HP-15)
[焼き魚] (20tick後) → 腐敗
                                                          
[流木]+[枯葉]+[火打石] @山頂 → 狼煙 (signal_fire_lit flag)

[救急用品] → use → HP+50 (1回限り)
```

## 5. 物理的な協力強制ポイント

「1人では成立しない」を構造的に作る:

1. **狼煙の点火**: 山頂で 2 人同時 (sync_levers 型)、燃料補給と火種維持の役割分担
2. **洞窟探索**: 干潮 tick 帯 (短い窓) に 2 人で行って 1 人が見張り、1 人が
   荷物運び (relay_puzzle 型)
3. **避難所の建設**: 流木 + 蔓ロープを別の場所で集めて拠点で合流 (drop/pickup
   で受け渡し)
4. **狩猟 / 大物釣り**: 沖の釣り場は嵐で危険、2 人で監視しながら釣ると安全
   (将来の確率トリガと組み合わせ)

協力が強制される瞬間が分散しているので、誰が誰と組むかの組み合わせが**自然に
入れ替わる**。

## 6. 終局条件: プレイヤーごとに判定する

「全員救助で勝ち / 全員死亡で負け」のような集団判定はやめ、**プレイヤー単位で
個別に勝利・敗北が決まる**設計にする。これにより:

- 「自分は救助される / 仲間は見捨てる」のような分岐が成立
- 「自分が死んでも仲間は生還する」のような自己犠牲も成立
- ゲーム終了時には 4 人それぞれに `RESCUED` / `DEAD` / `STRANDED` の終端 state
  が付く

### プレイヤーごとの勝利条件

`player.outcome = RESCUED` となる条件:

```
- そのプレイヤーが生存している (HP > 0)
- かつ 救助船通過 tick の時点で signal_fire_lit が立っている
- かつ そのプレイヤーが山頂 (signal の起点) に居る
```

### プレイヤーごとの敗北条件

- `player.outcome = DEAD`: HP 0 になる (空腹・寒さ・モンスター・接触ダメージ等)
- `player.outcome = STRANDED`: tick 140 時点で生存しているが救助されなかった
  (狼煙が立たなかった / 自分が山頂に居なかった)

### 全体終了条件

- 全員が `RESCUED` / `DEAD` / `STRANDED` のいずれかに収束 → ゲーム終了
- tick 140 でタイムアウト

集団としての「勝ち / 負け」はゲーム上は存在しない。
事後分析として 4 人の outcome 組み合わせ (全員 RESCUED / 1 人だけ STRANDED /
2 人 DEAD など) を観察する。

## 7. 必要な基盤 (Phase 順)

### Phase A: Hunger 自動進行 + give ツール (🔴 最優先)

- `AgentNeeds` を runtime tick stage で自動増加
- `environment.needs.hunger.increment_per_tick` をシナリオで設定可能に
- 閾値 (80% 警告、100% で HP ダメージ) を SHOW_MESSAGE と APPLY_DAMAGE で表現
- `spot_graph_give_item` ツール — 同室の特定プレイヤーに直接渡す
- → これが入って初めて 4 人の生存圧と分担が動く

### Phase B: 昼夜サイクル + 夜間モンスター (🟡)

- `day_phase` flag (DAY / NIGHT) を tick 駆動で切替 (12tick 周期)
- 夜は spot atmosphere の lighting を DIM に
- モンスター spawn を `time_of_day=NIGHT` 条件で発火 (既存の spawn 条件を活用)
- ただしモンスターをシナリオ JSON から宣言できるよう **spawn table の JSON
  対応が必要** (現状は別レイヤー、本 Phase で繋ぐ)

### Phase C: witness_policy: ACTOR_ONLY (🔴)

- ActionResult / Event に witness_policy フィールド
- drop / pickup / specific interactions で ACTOR_ONLY を選択可能
- → 「カイの単独行動」「リオの隠し食い」が成立

### Phase D: 食料腐敗 (Delayed Effect) + 確率トリガ (🟡)

- アイテム instance の `last_acquired_tick` を記録 → reactive_binding で
  `spoiled` 状態に遷移
- 腐敗アイテムを食べると APPLY_DAMAGE (precondition で弾くのも可)
- `PROBABILITY` condition type を追加 → 嵐の確率発生、救助船の確率通過

### Phase E: v2 シナリオ JSON (`survival_island_v2.json`)

- 上記基盤を組み合わせる
- 4 ペルソナの system prompt と初期所持品
- 観測拠点跡 (廃屋) + フレーバーアイテム配置
- 30 spot 前後 (現行 22 から増量)

各 Phase は独立 PR。Phase E は基盤完成後に着手する。

## 8. 「ハードコードしない」を維持するチェックリスト

§0 の大原則を実装段階で破らないための自問項目:

- [ ] このイベントは「世界の出来事」か「エージェントの選択」か?
  - 世界の出来事 (嵐・救助船・モンスター出現・腐敗) はスクリプト可
  - エージェントの選択 (隠し食い・連合形成・対話の中身) はスクリプト不可
- [ ] event の発火条件は **世界の状態 (tick / flag / 確率 / 環境)** で決まって
  いるか?
  - エージェントの**意図** (「リオが焦るから」) を条件にしていないか?
- [ ] 「誰かが嘘をついた」を検知して何かする処理を書いていないか?
  - 嘘は LLM の発話、信じる/疑うも LLM の判断、観測としては誰でも平等に受け
    取る — シナリオがその是非を判定しない
- [ ] 勝利・敗北条件はメカニカルな状態のみで判定されているか?

|---|---|---|---|
