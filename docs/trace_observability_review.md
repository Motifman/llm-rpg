# trace 観測性レビュー

この文書は、新しい実験機能を追加するときに「実験後にその機能が測れるか」を確認するためのレビュー観点です。実験設定の仕様ではなく、Claude Code / Codex などのコーディングエージェントに差分レビューさせるための作業メモです。

## 目的

実 LLM run は費用と時間がかかるため、機能を追加したあとに「trace を見ても効果を測れない」と分かる手戻りを避けます。

ここで守りたいのは、すべての機能に大きな計測基盤を足すことではありません。新機能が実験上の観察対象になるなら、その発火・入力・出力・失敗が trace から最低限たどれる状態にしてから run します。

## レビューの入口

次の変更を含む PR では、この文書を参照して観測性を確認します。

- エージェントの判断、記憶、目的、信念、想起、推論、行動選択に影響する変更
- 実験 profile から切り替える機能の追加
- trace 分析で見たい指標を増やす変更
- 既存 trace イベントの意味や payload を変える変更

UI 文言修正、ドキュメント修正、明らかに実験挙動へ影響しない内部整理では不要です。

## 確認すること

新しい実験機能を追加したら、PR 作成前に次を確認します。

- その機能が発火したことを trace から分かるか
- 効果判断に必要な値が trace payload に入っているか
- 機能が発火しなかった場合に、設定ミスなのか条件未成立なのかを trace から区別できるか
- 内部 metrics に届いた値が、必要なら trace へ転記されているか
- trace に出す値の粒度が、実験後の分析で使える粒度になっているか
- 観測点が増えたことを保証するテストがあるか

## レビューで聞く質問

Claude Code / Codex に差分レビューさせるときは、次の観点で見てもらいます。

```text
この PR を trace 観測性の観点でレビューしてください。
新しい実験機能、または既存実験機能の挙動変更がある場合、
その機能が発火したこと、効果判断に必要な値、失敗や未発火の理由が
trace から確認できるかを見てください。

不足がある場合は、どのファイルにどの trace イベントまたは payload を
追加すべきか、どのテストで守るべきかを指摘してください。
実験挙動に関係しない変更なら「観測性レビュー対象外」と明記してください。
```

## テストの考え方

単体テストでは、内部状態だけでなく観測の出口も確認します。

- buffer や store に入るだけでなく、対応する trace イベントが出ること
- 機能フラグや profile の組合せで、機能が黙って無効化されないこと
- metrics オブジェクトに入った値が、必要な trace payload に残ること
- 未発火が正常な条件分岐なら、その判断材料が trace から読めること

高コスト run を使って確認しません。まずは低コストの単体テスト、配線テスト、短い確認 run で確認します。

## やりすぎないための基準

次のものは、この文書の目的ではありません。

- すべての機能に新しい trace イベントを強制すること
- すべての実験で厳密な比較対象を要求すること
- 研究段階の仮説ごとに設定仕様を増やすこと
- trace payload を将来使うかもしれない値で膨らませること

必要なのは、「この run が終わったあと、見たかったことを見られるか」を PR 時点で確認することです。
