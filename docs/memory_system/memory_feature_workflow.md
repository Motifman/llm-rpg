# 記憶まわり機能の進め方（Git・レビュー・ブロッキング）

**実装計画の本文**（フェーズ P0〜P5・技術方針）は [episodic_memory_implementation_plan.md](./episodic_memory_implementation_plan.md)。  
本書はそのうち **手続き・ルール・ブランチの見方**だけを集約した参照用ドキュメントである。**記憶関連の作業を始めるとき・マージ前に毎回ここを確認する**。

---

## 1. Git・ブランチ・コミット

記憶まわりでブランチが巨大化しやすいため、次を合意として固定する。

1. **いま溜まっている変更**は、レビュー可能な単位で **main へマージして push する**（長寿命の巨大 feature ブランチを止める）。
2. **以降は機能単位ブランチ**にする。例: `feature/episodic-trace-spatial-fields`、`feature/episodic-rule-cues`、`feature/episodic-sqlite-index`。**1 ブランチに複数ドメインを詰め込まない**。
3. **コミットは小分け**にする。目安は「1 コミット＝レビューで説明できる 1 つの意図」（リネームだけ、テストだけ、ドキュメントだけ、なども分離してよい）。WIP をコミットに載せ続けない。
4. **コミットメッセージは「なぜ」を必ず書けるようにする。** 題名は Conventional Commits（`feat:` / `fix:` / `docs:` 等）。**本文に理由を一行以上**：防ぐバグ・再発防止・開ける次工程・避けたい負債など。将来 `git blame` と `git log` だけで意図が復元できることを優先する。
5. **GitHub へ PR は必ず `gh` で作る。** リモートへブランチを push したら、続けて **`gh pr create`**（または同等に **Pull Request が GitHub 上に存在する状態**）までを作業の一環とする。説明欄には §2.1 項 2 と同程度の内容（目的・コミット列・要約・レビュー観点）を書き、**今後の作業把握の単一の置き場**を GitHub に置く。`gh` 未インストール・認証不能は **§3 のブロッキング**。
6. PR も **小さく**し、マージ後に次ブランチを切る。

---

## 2. PR（`gh`）とサブエージェントによるマージ前レビュー

**PR は §1 項 5 のとおり必須**である。加えて、`main` へ入れる前に **サブエージェントによるレビュー**を挟む（公平な目線の確認。実装エージェントだけの判断で main に進めない）。**サブエージェントの起動に、利用者から実行の許可や「レビューして」を求めない。** 実装側エージェントがマージ判断の手前で **Cursor の Composer を用い、モデルは Composer 2 にして**別コンテキストのレビュアーとして起動する。

### 2.1 ルール（実装エージェント）

1. **マージ / push の前に**、**Composer 2** モデルで **Cursor の Composer** から **サブエージェント**（親チャットとは別コンテキストのレビュアー）を起動する。実装を済ませたエージェントと同一視点だけで main に入れない。**利用者の明示指示や実行許可を待たない**（コンテキストを渡した上で公平な視点を確保するため）。
2. 依頼文には **PR の説明欄に書く量**を入れる: **目的（なぜ）**、ブランチ名、**`main..ブランチ` のコミット一覧**（`git log`）、**変更の要約**（ファイル単位でよい）、**レビュー観点**（正しさ・境界条件・テスト不足・永続化・命名・層の責務など。プロジェクトの DDD ルールに触れてよい）。
3. **サブエージェントの結論（Approve / 要修正）と指摘リストが返るまで、後続のマージ判断に進まない**（結果を待たずに同一ターンで merge だけ完了させない）。
4. **サブエージェントのレビュー基準**: 正しさ・契約後方互換・セキュリティ・性能の致命的問題・回帰リスク・ドキュメントと実装の虚偽不一致など、**見過ごせない変更**（マージ前に直すか利用者決裁が要る類）が **1 件でも残る場合は Approve してはならない**。その場合の結論は **要修正** とし、ブロッカー／必須対応を列挙する。**指摘が軽微な改善提案や任意のテスト強化にとどまる**ときのみ **Approve** できる。
5. 結論を踏まえた次の動き: **要修正**なら指摘の反映に取り組む。**プロダクト判断・アーキ採否など利用者の決裁が必要な場合はそこで停止**し、ブロッキングがない変更・テスト・コミットだけは実装側で進めてよい。**Approve か対応完了後**、残タスクがなければ **`main` へのマージ**を行う（§1 のコミット規約に従う）。**マージは PR 経由を原則とし、`gh pr merge` または GitHub 上のマージでよい**（ローカル `git merge` のみで PR を閉じない運用は避ける）。
6. **指摘を反映するか、意図的に採用しない理由を PR 上またはコミット本文に短く残ってから**マージする。レビュー待ちのまま無言でマージしない。
7. 指摘対応は **小分けコミット**（§1 項 3・4）で積む。

### 2.2 補足

- レビューとマージを**同じ送信バッチ**に載せない（レビュー結果が返る前に merge だけ完了、を避ける）。
- **サブエージェントは Composer 2 を選んだ Composer で起動する**（本体チャットと別スレッド・別コンテキストのレビューとして使う想定）。
- **人間レビュアー**がいる場合は PR のレビュー要求とサブエージェントを併用してよい。いずれにせよ **見過ごせない問題が残る場合は Approve 禁止**（§2.1 項 4）を維持する。

## 3. ブロッキングとブランチの見方

作業を進める・マージする前に、**そのブランチから見たブロッキング**を明示する。次のいずれかに当てはまる場合は、実装を進めても **main へ進めない**（または利用者に確認する）。

| 観点 | 確認例 |
|------|--------|
| **リモート main** | `git fetch` 後、`main..HEAD` が正しい差分か。リモートが先行なら先に `pull` してからマージ判断。 |
| **別ブランチ・worktree** | 同一ファイル（例: `dtos.py`, `agent_orchestrator.py`）を別作業が触っていないか。触れているなら **合流順・コンフリクト解消**がブロッキングになりうる。 |
| **テスト・CI** | 関連 `pytest` 未実行、または CI 失敗はマージのブロッキング。 |
| **永続化・マイグレーション** | schema 変更とストア実装の不整合、ロールバック未定はブロッキングになりうる。 |
| **仕様・製品判断** | 計画上「ユーザ判断」の項目に触れる変更は、**決裁待ち**のブロッキング。 |
| **認証・push** | `git push` が環境要因で失敗している場合は、ローカルマージのみ完了していても **リモート未反映**としてブロッキングを残す。 |
| **PR（`gh`）** | 記憶まわりの変更で **GitHub 上に PR が無い**（`gh pr create` 未実施など）状態はマージ判断のブロッキング。 |

実装エージェントは、ターン終了時に **「残ブロッキングがあるか」** を一文で明示する（なければ「なし」）。

## 4. 直近のブランチ戦略（worktree・並列）

**目的**: 並列作業の可否を整理し、巨大ブランチと積み残しを防ぐ。**数月先の枝分かれ全部は決めない**。次の 1〜2 周回で足りる粒度に留める。

### 4.1 ブランチの切り方（例）

| ブランチの単位（例） | 含めるものの目安 | マージの仕方 |
|---------------------|------------------|--------------|
| `feature/episodic-trace-spatial` | Trace に構造化位置をコピー（P1） | main へ 1 PR。完了後に次へ。 |
| `feature/episodic-rule-cues` | ルール抽出 → `cues` のみ（P2） | 同上。P1 と干渉しなければ並列可。 |
| `feature/episodic-sqlite-index` | P4 の store・schema | P2 と別ワークツリーなら並列検討可。 |

### 4.2 並列化の条件

- **依存が薄い**（同ファイルを奪い合わない、DTO の破壊的変更が一方に偏らない）なら **git worktree** で別ディレクトリを生かし、**別エージェント／別チャットに明示的に委任**する。委任時は「ブランチ名・目的・触ってよいパス・合流条件（テストコマンド）」を渡す。
- **同一 DTO / orchestrator を両方が触る**場合は直列にするか、先に小さい PR で共有部分だけ main に入れる。

### 4.3 ワークツリー運用の注意

- 同じ venv を共有してよいが、**ビルド成果物や `.env` の上書き**に注意する。
- どの worktree がどのリモートブランチかを迷わないよう、**ディレクトリ名とブランチ名を対応**させる。
- **`worktrees/` 等のローカル作業ディレクトリ**を誤ってコミットしない（必要なら `.gitignore` を検討）。

### 4.4 先に決めておくとよいこと（短く）

- **コンフリクトが出やすい窪み**（`agent_orchestrator.py`, `dtos.py`）は「先に短い整備 PR」で触る回数を減らす。
- **SQLite schema 変更**を始めたらマイグレーション方針（バージョン番号・ロールバック）を一文だけでも残す。
- **レガシー episodic の削除**は利用箇所 grep → 実装計画の廃止表（§3.4）に行を足してから段階削除。

## 5. 作業完了後の記録（実装計画ドキュメントの更新コミット）

記憶まわりの**フェーズ完了・仕様寄りの方針変更・残タスクの棚卸し**は、実装を `main` に載せて終わりにせず、**[episodic_memory_implementation_plan.md](./episodic_memory_implementation_plan.md)** に反映する（該当セクションの ✅ / 査読表 / §3.4 など）。手続き・Git 運用の合意だけ変えるときは **本ファイル（memory_feature_workflow.md）** を更新する。

- **タイミング**: そのフェーズの実装がマージ可能になった判断がついた時点で、**同じ PR 内の最終コミット**でも、**直後の `docs:` コミット**でもよい。いずれにせよ **マージ前に計画ファイルが成果と矛盾していないこと**をブロッキング（§3 の「ドキュメントと実装の虚偽不一致」）とみなす。
- **コミット**: Conventional Commits で `docs:` を推奨。本文に「どの P / どの表を更新したか」を一行以上。

## 6. 記憶システムまわりのテストの書き方（推奨）

新規または **記憶（Passive / エンコード / v2 store）を広く触る変更**では、次を目安にする（既存テストの一括リライトは必須としない）。

1. **`pytest` クラスで機能単位をまとめる**  
   例: `class TestPassiveSubjectiveRecallComposer:`、`class TestEpisodicCueExtraction:`。ファイルが長くなるほど、**シナリオごとにクラスを分ける**と読みやすい。

2. **各テストメソッドに日本語の docstring**  
   **何を保証するか**（前提・期待・仕様との対応が分かる一行〜数行）を書く。英語名のメソッドだけに任せない。

3. **段階的適用**  
   手を入れたモジュールのテストからクラス化・docstring を足し、無関係なドメイン（例: sns）の既存ファイルは触らなくてよい。
