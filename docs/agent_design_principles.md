> 判断のうち、特定の機能ではなく **方針そのもの** を残す。

> に集約してある。本ファイルはその上位 — 個別判断を導くための哲学的
> ガードレール — を扱う。

> 生きたドキュメントなので、新しい原則が必要になったら追記する。
> 既存原則が間違っていたと判明したら、訂正履歴を残して書き換える。

## 1. エージェントとゲームの疎結合

**LLM に露出する経路には整数 ID を出さない**。人物・場所・物体すべて
名前で扱う。ツール引数は自然言語 / enum 文字列ベースのみ受ける。

なぜ:

- いつかエージェント部分を切り出して別ゲームに移植できる状態を保つため
- ID を渡す世界に慣れたエージェントは「ID で人物を識別する」前提で
  ペルソナを組んでしまい、剥がせなくなる
- ゲーム世界の概念は名前で十分。識別は内部で resolver が解決する

具体ルール:

- ツール引数: spot 名・人物名・物名は **文字列** で受ける
- prompt 露出: 「現在地と周囲」section も含めすべて名前 (= 既に基本そうなっている)
- 内部 cue (`EpisodicCue.value` の `spot_graph_player_42` など) は **内部 index 専用**。recall_text や prompt には漏らさない
- 例外: trace / log / debug 用途は ID を出して良い (= LLM が読まない経路)

破ってる箇所を見つけたら、その PR の責務とは別に「audit + fix」PR を切る。

## 2. ターン制ではなく観測駆動

このリポの LLM は **1 observation = 1 LLM 起動 = 1 tool 呼び出し** で動く。
通常の agentic loop の「tool × N → final message」は **無い**。

なぜ:

- ゲーム世界の時間進行とエージェントの意思決定を分離するため
- 観測が来ない間は LLM は休眠し、世界の経過に振り回されない
- LLM のコストを観測の重要度でゲートできる (= `schedules_turn=True` の観測のみ起動)

具体ルール:

- 「次の turn」という抽象は使わない。「次に observation が `schedules_turn=True` で起きるとき」と表現する
- gate 設計で「1 turn 1 回」と言いたくなったら、「1 起動 1 ツール呼び出し」が既に唯一の自然 unit なので、追加 gate は基本不要 (`tool_call_loop_guard` が同一引数連打を検知済)
- 時間表現は wall-clock の `datetime` (= observation の `occurred_at`) と game tick の 2 系統が並走する。主観時間ラベル ([docs/quality_checks/yesterday_v1_baseline.md](./quality_checks/yesterday_v1_baseline.md) 経由) は前者を使う

## 3. 質感は機能の総和ではない

数値テスト pass = 質感改善 ではない。各 PR で「振る舞いの質感がどう
変わったか」を 1 段落で記録する場 ([docs/quality_checks/](./quality_checks/))
を持つ。機能の集合に終わらせず、**創発を目指す**。

なぜ:

- 「ハードコードされた機能の集合」と「人間らしさが結果として滲み出る
  エージェント」は別物
- 機能で埋める対症療法は別の場所に同じ穴を生む
- 質感の点検場が無いと「ユニットテスト全部緑、でも何も変わってない」が
  ずっと続く

具体ルール:

- 質感シナリオ (= LLM を呼ばずに prompt 構造を見る pytest テスト) を
  シリーズごとに 1 つ用意する
- 各 PR の DoD として「質感ドキュメントの末尾に 1 段落追記」を入れる
  (= 軽さは絶対に守る、自動化しない)
- ネガティブ所見 (= 「変わってない」) こそ価値がある

## 4. 自己の継続性・脆弱性

エージェントが **「未来の自分は存在する」「行動は取り返しがつかない」**
と内部モデルで持てるかが、価値づけ・約束・恐れ・喜びの根。これが無いと、
感情ラベルを足してもただの装飾になる。

なぜ:

- valuation / care / concern は機能ではなく **主体の継続性の帰結** として
  生まれる
- 「失敗しても初期化で戻る」感覚があるエージェントは、どれだけ感情
  フィールドを持たせても「うわっ怖い」を演じるだけになる
- Issue #526 の不在 0 (= 5 つの不在の根) として位置付けている

具体ルール:

- 永続化 (snapshot / episode store / memory link) は「未来の自分への
  申し送り」として設計する。揮発性に倒さない
- 「取り返しのつかない損失」(= 死、別離、約束破り、信頼喪失) を後付け
  でなく最初から世界の前提に組み込む
- 「過去の自分が考えていたこと」を能動的に取り戻せる経路を持つ
  (= `memory_recall` 等)

## 5. 失敗の質感を許容する

「思い出せない」「うろ覚え」「誤想起」の階層を agent に経験させる。
完璧に思い出させようとせず、**人間が経験する failure modes** を再現する。

なぜ:

- 完璧に思い出せる agent は不気味の谷に落ちる
- 「思い出せない」を知っている agent は、「思い出そうとする」を意志する
  動機を持つ (= 不在 2 agent-driven 想起への接続)
- "0 件 vs 1 件以上" の二値で終わらせず、「うっすら」「曖昧」「全く
  浮かばない」を表現する

具体ルール:

- 検索系ツールは空結果でも特別な失敗扱いにせず、「思い出そうとしたが
  何も浮かばなかった」を自然な response として返す
- 古い episode / cue マッチ弱の結果は「うっすら」風に質感を付ける余地を
  残す (= v0 では二値で OK、後続で 3 段階)
- 「うろ覚えなのに自信満々」を防ぐため、ツール description で誤想起
  リスクを LLM に明示する

## 6. silent failure を構造的に避ける

例外を握りつぶさない、空 list で fallback したら必ず log を残す。
**失敗を後から発見できる構造を維持する**。

なぜ:

- silent failure は「動いてるように見えて壊れてる」を生み、回帰検出を
  数週間遅らせる
- 構造的な発見可能性 (= warning 級ログ、trace event) を諦めると、機能を
  足すたびに見えない断絶が増える

具体ルール:

- `try/except: pass` は禁止。常に `_logger.warning(... exc_info=True)` か
  類似で痕跡を残す
- fallback で空 list / None を返すとき、その fallback path に乗ったことが
  あとから grep / trace で見える状態にする
- 詳細は [docs/design_decisions.md #5 / #8 / #14 / #18](./design_decisions.md)

## 7. 他エージェントから何が見えるか / 見えないかを設計に乗せる

内部認知ツール (= `memory_recall` 等) でも、他者から見て「考え込んで
いる」が表現される余地を将来残す。**完全に internal で終わるツールは
「動いていない」と等価**。それは現実的に正しいか毎回問う。

なぜ:

- 人間は「考えること」も身体動作として外から見える (= 動きが止まる、
  目が泳ぐ、返事が遅れる)
- 完全 internal な行為があると、社会的 dynamics が痩せる (= 「あいつ
  何考えてるんだろう」の余地が消える)
- ただし v0 から全部入れる必要はない。`A: internal only → B: 時間消費 →
  C: 観測発生` の段階で追加できる構造にしておく

具体ルール:

- 新しい内部認知ツールは「observation を発生させない」をデフォルトに、
  あとから hook を足せる構造にする
- 「このツールは他者から見て何に見えるか」を tool 設計時に 1 行書く
