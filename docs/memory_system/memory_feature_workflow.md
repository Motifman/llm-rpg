# 複数エージェント作業の進め方（Git・レビュー・ブロッキング）

本書は、複数のエージェント／チャット／ブランチで作業するときの **手続き・ルール・ブランチの見方**を集約した参照用ドキュメントである。対象が記憶システムに限られなくても、並列実装・PR 分割・レビューを行う作業では毎回ここを確認する。

対象機能に個別の計画ファイルがある場合は、その計画も必ず読む。例: エピソード記憶では [episodic_memory_implementation_plan.md](./episodic_memory_implementation_plan.md)。

## 0. 結論: 必須手順

複数エージェントでの実装・修正・計画更新では、次を **この順番で必ず実施する**。

1. **このファイルと対象機能の計画を読む**  
   作業開始前に本ファイルを読み、機能別の plan / spec / workflow があればそれも読む。
2. **渡す側も受け取る側も、作業ディレクトリが専用 git worktree か確認する**  
   並列作業、別チャット、別エージェント、サブエージェント相当の実装担当に渡す作業は、同じ作業ディレクトリで進めない。担当エージェントは、開始時に自分の `cwd` が専用 worktree であり、ブランチ名と一致していることを確認する。専用 worktree ではない場合は作業を止め、オーケストレーターへ worktree 作成を依頼する。
3. **機能単位ブランチにする**  
   1 ブランチに複数ドメインを詰め込まない。ブランチ名・目的・触ってよいパス・合流条件を明示する。
4. **小分けコミットにする**  
   1 コミットはレビュー可能な 1 つの意図にする。WIP や仮実装を積まない。
5. **関連テストを実行する**  
   未実行テスト、失敗テスト、CI 失敗はマージのブロッキングとして扱う。
6. **PR 作成前に Composer 2 の別エージェントでレビューする**  
   利用者の許可待ちは不要。担当エージェントはレビュー依頼を出し、Approve / 要修正の結論と指摘リストが返るまで待つ。
7. **レビュー結果を反映してから `gh pr create` で PR を作る**  
   要修正なら対応して再確認する。Approve または対応完了後にだけ、GitHub 上に PR がある状態まで進める。PR 説明にはレビュー結果と対応有無を必ず書く。
8. **ブロッキングを明示する**  
   ターン終了時または PR 説明で、残ブロッキングがあるかを必ず書く。
9. **実装と計画が矛盾しないよう更新する**  
   フェーズ完了・方針変更・省略したものは、対象機能の計画ファイルに反映する。

特に **別エージェントへ渡す実装作業で worktree を使わないことは、このワークフロー違反**である。委任された側も「専用 worktree ではない」と気づいた時点で止まる。サブエージェント自身が worktree を作れない場合は、オーケストレーターが先に作ってから委任する。

## 1. Git・ブランチ・コミット

複数エージェント作業ではブランチが巨大化しやすいため、次を合意として固定する。

1. **いま溜まっている変更**は、レビュー可能な単位で **main へマージして push する**（長寿命の巨大 feature ブランチを止める）。
2. **以降は機能単位ブランチ**にする。例: `feature/foo-model`、`feature/foo-store`、`feature/foo-ui`。**1 ブランチに複数ドメインを詰め込まない**。
3. **コミットは小分け**にする。目安は「1 コミット＝レビューで説明できる 1 つの意図」（リネームだけ、テストだけ、ドキュメントだけ、なども分離してよい）。WIP をコミットに載せ続けない。
4. **コミットメッセージは「なぜ」を必ず書けるようにする。** 題名は Conventional Commits（`feat:` / `fix:` / `docs:` 等）。**本文に理由を一行以上**：防ぐバグ・再発防止・開ける次工程・避けたい負債など。将来 `git blame` と `git log` だけで意図が復元できることを優先する。
5. **GitHub へ PR は必ず `gh` で作る。** ただし PR 作成前に §2 の Composer 2 レビューを完了させる。リモートへブランチを push したら、レビュー結果を待ってから **`gh pr create`**（または同等に **Pull Request が GitHub 上に存在する状態**）までを作業の一環とする。説明欄には §2.1 項 2 と同程度の内容（目的・コミット列・要約・レビュー観点・レビュー結果）を書き、**今後の作業把握の単一の置き場**を GitHub に置く。`gh` 未インストール・認証不能は **§3 のブロッキング**。
6. PR も **小さく**し、マージ後に次ブランチを切る。

## 2. PR（`gh`）とサブエージェントによる PR 作成前レビュー

**PR は §1 項 5 のとおり必須**である。ただし、PR 作成前に **サブエージェントによるレビュー**を挟む（公平な目線の確認。実装エージェントだけの判断で PR に進めない）。**サブエージェントの起動に、利用者から実行の許可や「レビューして」を求めない。** 実装側エージェントは PR 作成の手前で **Cursor の Composer を用い、モデルは Composer 2 にして**別コンテキストのレビュアーとして起動する。マージ可否は担当エージェントが決めない。PR 作成後、オーケストレーターまたは人間が CI・レビュー・依存関係を見て判断する。

ここでの **Approve は PR 作成前レビューの合格**であり、GitHub PR のマージ承認ではない。

### 2.1 ルール（実装エージェント）

1. **PR 作成前に**、**Composer 2** モデルで **Cursor の Composer** から **サブエージェント**（親チャットとは別コンテキストのレビュアー）を起動する。実装を済ませたエージェントと同一視点だけで PR に進まない。**利用者の明示指示や実行許可を待たない**（コンテキストを渡した上で公平な視点を確保するため）。
2. 依頼文には **PR の説明欄に書く量**を入れる: **目的（なぜ）**、ブランチ名、worktree パス、**`main..ブランチ` のコミット一覧**（`git log`）、**変更の要約**（ファイル単位でよい）、**レビュー観点**（正しさ・境界条件・テスト不足・永続化・命名・層の責務など。プロジェクトの DDD ルールに触れてよい）。
3. **サブエージェントの結論（Approve / 要修正）と指摘リストが返るまで、PR を作成しない**（結果を待たずに同一ターンで PR 作成・merge へ進めない）。
4. **サブエージェントのレビュー基準**: 正しさ・契約後方互換・セキュリティ・性能の致命的問題・回帰リスク・ドキュメントと実装の虚偽不一致など、**見過ごせない変更**（マージ前に直すか利用者決裁が要る類）が **1 件でも残る場合は Approve してはならない**。その場合の結論は **要修正** とし、ブロッカー／必須対応を列挙する。**指摘が軽微な改善提案や任意のテスト強化にとどまる**ときのみ **Approve** できる。
5. 結論を踏まえた次の動き: **要修正**なら指摘の反映に取り組む。**プロダクト判断・アーキ採否など利用者の決裁が必要な場合はそこで停止**し、ブロッキングがない変更・テスト・コミットだけは実装側で進めてよい。**Approve か対応完了後**、残タスクがなければ PR を作成する。
6. **指摘を反映するか、意図的に採用しない理由を PR 説明またはコミット本文に短く残す**。レビュー待ちのまま無言で PR 作成・マージしない。
7. 指摘対応は **小分けコミット**（§1 項 3・4）で積む。

### 2.2 補足

- レビューと PR 作成を**同じ送信バッチ**に載せない（レビュー結果が返る前に PR だけ作る、を避ける）。
- **サブエージェントは Composer 2 を選んだ Composer で起動する**（本体チャットと別スレッド・別コンテキストのレビューとして使う想定）。
- **人間レビュアー**がいる場合は PR のレビュー要求とサブエージェントを併用してよい。いずれにせよ **見過ごせない問題が残る場合は Approve 禁止**（§2.1 項 4）を維持する。

## 3. ブロッキングとブランチの見方

作業を進める・マージする前に、**そのブランチから見たブロッキング**を明示する。次のいずれかに当てはまる場合は、実装を進めても **main へ進めない**（または利用者に確認する）。

| 観点 | 確認例 |
|------|--------|
| **リモート main** | `git fetch` 後、`main..HEAD` が正しい差分か。リモートが先行なら先に `pull` してからマージ判断。 |
| **別ブランチ・worktree** | 同一ファイル（例: `dtos.py`, `agent_orchestrator.py`）を別作業が触っていないか。触れているなら **合流順・コンフリクト解消**がブロッキングになりうる。 |
| **テスト・CI** | 関連 `pytest` 未実行、または CI 失敗はマージのブロッキング。 |
| **永続化・マイグレーション** | schema 変更とストア実装の不整合、ロールバック未定はブロッキングになりうる。 |
| **仕様・製品判断** | 計画上「ユーザ判断」の項目に触れる変更は、**決裁待ち**のブロッキング。 |
| **認証・push** | `git push` が環境要因で失敗している場合は、ローカルマージのみ完了していても **リモート未反映**としてブロッキングを残す。 |
| **PR（`gh`）** | 作業ブランチに対して **GitHub 上に PR が無い**（`gh pr create` 未実施など）状態はマージ判断のブロッキング。 |

実装エージェントは、ターン終了時に **「残ブロッキングがあるか」** を一文で明示する（なければ「なし」）。

## 4. 直近のブランチ戦略（worktree・並列）

**目的**: 並列作業の可否を整理し、巨大ブランチと積み残しを防ぐ。**数月先の枝分かれ全部は決めない**。次の 1〜2 周回で足りる粒度に留める。

**必須ルール**: 並列作業・別チャット・別エージェント委任では、必ず git worktree を使う。同じ作業ディレクトリで複数エージェントが同時に実装してはならない。worktree ディレクトリ名は、原則としてブランチ名をファイルシステム向けに置換したものにする（例: `feature/foo-store` → `worktrees/feature-foo-store`）。この規則により、オーケストレーターが後でまとめて削除しやすくする。

### 4.1 ブランチの切り方（例）

| ブランチの単位（例） | 含めるものの目安 | マージの仕方 |
|---------------------|------------------|--------------|
| `feature/foo-model` | 共通 DTO / value object / 契約 | main へ 1 PR。完了後に依存ブランチへ進む。 |
| `feature/foo-store` | store / repository / 永続化 | 同上。model と干渉しなければ並列可。 |
| `feature/foo-ui` | presentation / UI / demo | 契約が安定してから合流。 |

### 4.2 並列化の条件

- **依存が薄い**（同ファイルを奪い合わない、DTO の破壊的変更が一方に偏らない）場合のみ並列化してよい。並列化する場合は **必ず git worktree** で別ディレクトリを作り、**別エージェント／別チャットに明示的に委任**する。委任時は「worktree のパス・ブランチ名・目的・触ってよいパス・合流条件（テストコマンド）」を渡す。担当エージェントは開始時にその worktree にいることを確認し、違う場合は作業を止める。
- **同一 DTO / orchestrator を両方が触る**場合は直列にするか、先に小さい PR で共有部分だけ main に入れる。

### 4.3 ワークツリー運用の注意

- 同じ venv を共有してよいが、**ビルド成果物や `.env` の上書き**に注意する。
- どの worktree がどのリモートブランチかを迷わないよう、**ディレクトリ名とブランチ名を対応**させる。
- **`worktrees/` 等のローカル作業ディレクトリ**を誤ってコミットしない（必要なら `.gitignore` を検討）。

### 4.4 先に決めておくとよいこと（短く）

- **コンフリクトが出やすい窪み**（共通 DTO、orchestrator、wiring、schema など）は「先に短い整備 PR」で触る回数を減らす。
- **SQLite schema 変更**を始めたらマイグレーション方針（バージョン番号・ロールバック）を一文だけでも残す。
- 既に削除した旧実装を再導入するときは、旧 API に戻さず、対象機能の計画・spec 起点で新規に作る。

## 5. 作業完了後の記録（計画ドキュメントの更新コミット）

フェーズ完了・仕様寄りの方針変更・残タスクの棚卸しは、実装を `main` に載せて終わりにせず、対象機能の計画ファイルに反映する。手続き・Git 運用の合意だけ変えるときは本ファイルを更新する。

- **タイミング**: そのフェーズの実装がマージ可能になった判断がついた時点で、**同じ PR 内の最終コミット**でも、**直後の `docs:` コミット**でもよい。いずれにせよ **マージ前に計画ファイルが成果と矛盾していないこと**をブロッキング（§3 の「ドキュメントと実装の虚偽不一致」）とみなす。
- **コミット**: Conventional Commits で `docs:` を推奨。本文に「どの計画・どの残タスクを更新したか」を一行以上。

## 6. テストの書き方（推奨）

新規機能や共有部を広く触る変更では、次を目安にする（既存テストの一括リライトは必須としない）。

1. **`pytest` クラスで機能単位をまとめる**  
   例: `class TestFooStore:`、`class TestFooRules:`。ファイルが長くなるほど、**シナリオごとにクラスを分ける**と読みやすい。

2. **各テストメソッドに日本語の docstring**  
   **何を保証するか**（前提・期待・仕様との対応が分かる一行〜数行）を書く。英語名のメソッドだけに任せない。

3. **段階的適用**  
   手を入れたモジュールのテストからクラス化・docstring を足し、無関係なドメイン（例: sns）の既存ファイルは触らなくてよい。
